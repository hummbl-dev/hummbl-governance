# HUMMBL Init Standard v0.1

**Status:** v0.1
**Steward:** HUMMBL Research Institute
**Approving human:** Reuben Bowlby
**Source of record:** git
**Depends on:** HUMMBL Repo Standard v0.1

## 1. Purpose

This standard canonizes the approach to initializing every class of artifact
in the HUMMBL fleet. It exists because "just create a repo" and "just add
governance" have produced inconsistent results across 91+ repos. This
standard makes initialization deterministic, auditable, and repeatable.

## 2. Scope

This standard covers the initialization of:

- Repositories (new and existing)
- Governance artifact stacks (CONSTITUTION, KRINEIA, manifest, CODEOWNERS)
- Python packages (pyproject.toml, package structure, tests)
- TypeScript packages (package.json, tsconfig, package structure)
- Cloudflare Workers (wrangler.toml, bindings, env)
- MCP servers (entry points, tool registration)
- Skills (SKILL.md, evals, routing)
- ADRs (numbering, structure, acceptance)
- Receipts (genesis, chaining, verification)
- Local directories (git init, .gitignore, README)
- Fleet inventory entries (adding to the canonical inventory)

## 3. Initialization principles

### 3.1 Determinism

Every initialization produces the same file set for the same artifact class.
No agent improvises the file list. The file set is defined in this standard
and implemented by a generator script.

### 3.2 Grounding

Initialization files are grounded in the repo's actual content. A
CONSTITUTION.md's protected invariants are derived from the repo's README,
AGENTS.md, existing CI, and dependency manifest -- not from a generic
template.

### 3.3 Receipt-first

Every initialization that creates or modifies normative files produces a
KRINEIA genesis receipt. The receipt is written before the PR is created.

### 3.4 No blind templating

The HUMMBL Repo Standard explicitly forbids blind templating. Every file
generated by initialization must be reviewed for repo-specific accuracy
before merge. The generator produces a draft; the human or reviewing agent
confirms.

### 3.5 Idempotency

Running the initializer on an already-initialized repo is a no-op. It does
not overwrite existing files. It reports what exists and what is missing.

### 3.6 Single source of truth

The canonical standard documents live in `hummbl-dev/hummbl-governance`.
Templates live in `hummbl-dev/.github/governance/templates/`. The generator
script lives in `hummbl-dev/hummbl-governance/tools/`. No other repo holds
authoritative copies.

## 4. Initialization procedures

### 4.1 Repository initialization

**Trigger:** A new repo is created, or an existing repo needs the governance
stack.

**Steps:**

1. **Classify the repo.**
   - `active` vs `archived`
   - `fork` vs `non-fork`
   - `public` vs `private`
   - `python` vs `typescript` vs `markdown-only` vs `mixed`

2. **Determine the file set.**

   | Artifact | Active non-fork | Archived fork | Archived non-fork |
   |----------|-----------------|---------------|-------------------|
   | `CONSTITUTION.md` | Yes | No (use HUMMBL_FORK.md) | Yes |
   | `KRINEIA.md` | Yes | Yes | Yes |
   | `hummbl.repo.yaml` | Yes | Yes | Yes |
   | `CODEOWNERS` | Yes | No | Yes |
   | `AGENTS.md` | Yes (if missing) | No | Yes (if missing) |
   | `_receipts/krineia/primary.jsonl` | Yes | Yes | Yes |
   | `docs/adr/ADR-001-*` | Yes | No | Yes |
   | `docs/handoffs/` | Yes | No | Yes |
   | `HUMMBL_FORK.md` | No | Yes | No |

3. **Ground the invariants.**
   - Fetch README.md, AGENTS.md, pyproject.toml, package.json from the repo.
   - Derive protected invariants from actual content (stdlib-only, test gate,
     contracts canonical, etc.).
   - Minimum 3 invariants. Maximum 8 (more becomes unmanageable).

4. **Generate files.**
   - Use the generator script (`tools/init_repo.py` — to be built).
   - Each file is written to disk with repo-specific content.

5. **Generate genesis receipt.**
   - SHA-256-chained, append-only, canonical JSON.
   - `prev_hash` = `"0" * 64` for genesis.
   - Event: `governance.manifest_adopted`.

6. **Commit, push, PR.**
   - Branch: `feat/devin/repo-governance-baseline`
   - Commit: Conventional Commits format, GPG-signed.
   - PR body: file list, invariant count, test plan.

7. **Merge.**
   - Admin merge for docs-only governance files.
   - CI must be green if the repo has CI that triggers on the changed paths.

8. **Verify.**
   - Confirm files exist on the default branch via API.
   - Confirm repo is re-archived if it was archived.

### 4.2 Python package initialization

**Trigger:** A new Python package is created within a repo.

**File set:**

```
pyproject.toml          # [project] with name, version, dependencies=[], requires-python=">=3.11"
src/<package>/          # package directory
src/<package>/__init__.py  # version string only
tests/__init__.py       # empty
tests/test_<package>.py # smoke test (import + version check)
README.md               # if missing
```

**pyproject.toml template:**

```toml
[project]
name = "<package-name>"
version = "0.1.0"
description = "<description>"
requires-python = ">=3.11"
dependencies = []

[project.optional-dependencies]
test = ["pytest>=7.0", "pytest-cov>=4.0"]

[tool.setuptools.packages.find]
include = ["<package_name>*"]
```

**Rules:**
- `dependencies = []` is the default. Adding a runtime dependency requires
  an ADR documenting the justification.
- `requires-python = ">=3.11"` is the floor.
- Test deps go in `[project.optional-dependencies.test]` only.
- Coverage threshold: 80% (enforced in CI).

### 4.3 TypeScript package initialization

**Trigger:** A new TypeScript package is created.

**File set:**

```
package.json            # name, version, type: "module"
tsconfig.json           # strict mode
src/index.ts            # entry point (named exports)
tests/<name>.test.mjs   # Node built-in test runner (NOT .test.ts)
README.md               # if missing
```

**package.json template:**

```json
{
  "name": "<package-name>",
  "version": "0.1.0",
  "type": "module",
  "dependencies": {},
  "devDependencies": {
    "typescript": "^5",
    "@types/node": "^20"
  }
}
```

**Rules:**
- `dependencies: {}` is the default. Adding a runtime dependency requires
  justification in the PR.
- Test files use `.test.mjs` extension only (enforced by lint script).
- `tsconfig.json` uses `strict: true`.
- Named exports only (no default exports).

### 4.4 Cloudflare Worker initialization

**Trigger:** A new Cloudflare Worker is created.

**File set:**

```
wrangler.toml           # name, main, compatibility_date, bindings
src/index.ts            # fetch handler
tests/index.test.mjs    # vitest or node:test
package.json            # devDeps: wrangler, @cloudflare/workers-types
```

**wrangler.toml template:**

```toml
name = "<worker-name>"
main = "src/index.ts"
compatibility_date = "2026-06-01"

# [[d1_databases]]
# binding = "DB"
# database_name = "<name>"
# database_id = "<id>"

# [[kv_namespaces]]
# binding = "KV"
# id = "<id>"
```

**Rules:**
- Bindings are declared in `wrangler.toml`, not hardcoded.
- Secrets use `wrangler secret put`, never committed.
- `compatibility_date` is set to the creation date.
- The Worker must have a health check endpoint (`GET /health`).

### 4.5 MCP server initialization

**Trigger:** A new MCP server is added to hummbl-governance.

**File set:**

```
hummbl_governance/mcp_<name>.py    # server module
tests/test_mcp_<name>.py           # test suite
```

**Entry point in pyproject.toml:**

```toml
[project.scripts]
hummbl-<name>-mcp = "mcp_<name>:main"
```

**Rules:**
- Stdlib-only (no `mcp` package dependency — use raw JSON-RPC over stdio).
- Each tool has a JSON Schema for arguments.
- Server registers tools in a `TOOLS` dict at module level.
- Tests cover: tool discovery, argument validation, response shape.

### 4.6 Skill initialization

**Trigger:** A new skill is created.

**File set:**

```
.devin/skills/<name>/SKILL.md       # skill definition
.devin/skills/<name>/evals/         # eval prompts (if applicable)
```

**SKILL.md structure:**

```markdown
# <skill-name>

<one-line description>

## Trigger Context
<when to use this skill>

## Workflow
<numbered steps>

## Output Templates
<structured output format>

## Guardrails
<constraints and non-actions>

## Skill Chains
<upstream and downstream chains>
```

**Rules:**
- Skill names are lowercase, hyphenated.
- One skill per directory.
- Eval prompts live in `evals/` subdirectory.
- Skills are validated by `skill-test` before merge.

### 4.7 ADR initialization

**Trigger:** A decision needs to be recorded.

**File:**

```
docs/adr/ADR-<number>-<kebab-title>.md
```

**Structure:**

```markdown
# ADR-<number> — <title>

- **Status:** proposed | accepted | superseded | deprecated
- **Date:** YYYY-MM-DD
- **Decision owner:** <name>
- **Steward:** HUMMBL Research Institute
- **Supersedes:** <ADR-number or none>

## Context
<why this decision is needed>

## Decision
<what was decided>

## Alternatives considered
<what else was on the table>

## Consequences
- **Positive:** <benefits>
- **Negative:** <costs>

## Receipts
<KRINEIA receipt references>
```

**Numbering:**
- Per-repo, sequential, zero-padded to 3 digits.
- `ADR-001` is always the repo governance baseline.
- Superseded ADRs keep their number; the superseding ADR references it.

### 4.8 Receipt initialization

**Trigger:** A governance event occurs (manifest adoption, invariant change,
authority change, release).

**File:** `_receipts/krineia/primary.jsonl` (append-only)

**Receipt structure:**

```json
{
  "id": "<uuid4>",
  "time": "<ISO 8601 Z>",
  "state": {
    "event": "<event.type>",
    "payload": { ... }
  },
  "drift": null,
  "prev_hash": "<sha256 of previous receipt's canonical form, or 64 zeros for genesis>",
  "hash": "<sha256 of this receipt's canonical form>"
}
```

**Canonical form:** `json.dumps(receipt_without_hash, sort_keys=True, separators=(",", ":"))`

**Rules:**
- Append-only. Never edit or delete.
- `prev_hash` chains to the previous receipt's `hash`.
- Genesis receipt has `prev_hash = "0" * 64`.
- The `hash` field is computed last, from the canonical form of the receipt
  without the `hash` field.
- Receipts are verified by `tools/verify_chain.py` (in krineia repo).

### 4.9 Local directory initialization

**Trigger:** A new local directory is created for development.

**Steps:**

1. `mkdir <dir>`
2. `cd <dir>`
3. `git init`
4. Create `.gitignore` (deny-by-default for repos in `$HOME`):
   ```
   /*
   !.gitignore
   !README.md
   !AGENTS.md
   ```
5. Create `README.md` with one-line description.
6. If the directory will become a repo, follow §4.1.

**Rules:**
- Local directories in `~/PROJECTS/` are expected to have `.git`.
- Local directories in `~/` (home) that are not git repos are classified
  as "non-git content" and tracked in the fleet inventory.
- Non-git directories should be archived or promoted to git repos.

### 4.10 Fleet inventory entry initialization

**Trigger:** A new repo or significant local directory is created.

**Steps:**

1. Add an entry to `docs/standards/FLEET_INVENTORY_<date>.md` in
   `hummbl-dev/hummbl-governance`.
2. Classify the entry by surface (private GitHub+Gitea, public GitHub,
   Gitea-only, local git, local non-git, gitignored).
3. Record: name, description, language, dependencies, file count.
4. Commit with `docs(governance): add <name> to fleet inventory`.

## 5. Initialization order

When initializing a new repo from scratch, follow this order:

1. Create the repo on GitHub (and Gitea mirror if private).
2. Clone locally.
3. Initialize the package (§4.2 for Python, §4.3 for TypeScript).
4. Initialize the governance stack (§4.1).
5. Initialize CI (if applicable).
6. Initialize the fleet inventory entry (§4.10).
7. Create the initial PR.
8. After merge, verify files on the default branch.

## 6. Generator script

The canonical generator script will live at:
`hummbl-dev/hummbl-governance/tools/init_repo.py`

It will:
- Accept a repo name and classification.
- Fetch repo metadata (README, AGENTS.md, pyproject.toml, package.json).
- Derive repo-specific invariants.
- Generate the full file set.
- Generate the genesis receipt.
- Stage, commit, push, and create a PR.

**Status:** Not yet built. The current rollout used ad-hoc Python scripts
in `/tmp/hummbl-rollout/`. The generator will consolidate these into a
single deterministic tool.

## 7. Validation

Every initialization must be validated:

| Check | Method |
|-------|--------|
| Files exist on default branch | `gh api repos/.../contents/<file>` |
| Genesis receipt passes hash chain | `python tools/verify_chain.py` |
| `hummbl.repo.yaml` is valid YAML | `python -c "import yaml; yaml.safe_load(open('hummbl.repo.yaml'))"` |
| CONSTITUTION.md has >= 3 invariants | `grep -c '^\d\.' CONSTITUTION.md` |
| CODEOWNERS has default rule | `grep '^\*' CODEOWNERS` |
| ADR is in `docs/adr/` | `ls docs/adr/` |
| PR was merged | `gh pr view <num> --json state` |

## 8. Amendment

Changes to this standard require: a PR, an ADR, a KRINEIA receipt, and
human approval (Reuben Bowlby).
